# Docker Deployment

VxLLM provides Docker images for the server and voice service, orchestrated with Docker Compose.

## Quick Start

Pull from Docker Hub and run:

```bash
docker pull datahase/vxllm
docker run -p 11500:11500 datahase/vxllm
```

The server will be available at `http://localhost:11500`.

### Using Docker Compose

For the full stack (server + voice service), create a `docker-compose.yml`:

```yaml
services:
  server:
    image: datahase/vxllm:latest
    ports:
      - "11500:11500"
    environment:
      HOST: "0.0.0.0"
      VOICE_URL: "http://voice:11501"
      DATABASE_URL: "file:/data/local.db"
      MODELS_DIR: "/data/models"
      API_KEY: "${API_KEY:-}"
      CORS_ORIGINS: "${CORS_ORIGINS:-*}"
    volumes:
      - vxllm-data:/data
    depends_on:
      - voice

  voice:
    image: datahase/vxllm-voice:latest
    ports:
      - "11501:11501"
    environment:
      VOICE_HOST: "0.0.0.0"
      MODELS_DIR: "/data/models"
    volumes:
      - vxllm-data:/data

volumes:
  vxllm-data:
```

Then run:

```bash
docker compose up -d
```

## Architecture

Docker Compose starts two services:

| Service | Port | Description |
|---------|------|-------------|
| `server` | 11500 | Bun/Hono server with LLM inference |
| `voice` | 11501 | Python voice service for STT/TTS |

Both services share a `vxllm-data` Docker volume for the SQLite database and downloaded models.

## Configuration

Edit `docker/.env` to customize your deployment:

```bash
# API key for authentication (required for remote access)
API_KEY=vx-your-secret-key-here

# Server port
PORT=11500

# Voice service port
VOICE_PORT=11501

# Allowed CORS origins
CORS_ORIGINS=https://your-frontend.example.com
```

### Authentication

When running in Docker, the server binds to `0.0.0.0`, which enables authentication. Set an `API_KEY` in your `.env` file:

```bash
API_KEY=vx-your-secret-key-here
```

Clients must include this key in requests:

```bash
curl http://your-server:11500/v1/chat/completions \
  -H "Authorization: Bearer vx-your-secret-key-here" \
  -H "Content-Type: application/json" \
  -d '{"model": "qwen2.5:7b", "messages": [{"role": "user", "content": "Hello"}]}'
```

## Pulling Models

After starting the containers, pull a model:

```bash
curl http://localhost:11500/api/models/pull \
  -H "Content-Type: application/json" \
  -d '{"name": "qwen2.5:7b"}'
```

Models are stored in the shared Docker volume and persist across container restarts.

## Building from Source

If you prefer to build the Docker images yourself instead of pulling from Docker Hub:

```bash
git clone https://github.com/vxllm/vxllm.git
cd vxllm
docker compose -f docker/docker-compose.yml up -d
```

The `docker/docker-compose.yml` in the repository uses `build` directives to compile from source. This is useful for development or customization.

## GPU Support

### NVIDIA GPU

To use NVIDIA GPU acceleration in Docker, install the [NVIDIA Container Toolkit](https://docs.nvidia.com/datacenter/cloud-native/container-toolkit/install-guide.html) and add GPU configuration to the server service:

```yaml
services:
  server:
    # ... existing config ...
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]
```

### Apple Silicon

Docker on Apple Silicon does not support Metal GPU passthrough. For GPU acceleration on macOS, run VxLLM natively (CLI or desktop app) instead of Docker.

## Data Management

### Volumes

The `vxllm-data` volume stores:
- SQLite database (`/data/local.db`)
- Downloaded models (`/data/models/`)

### Backup

```bash
# Create a backup of the data volume
docker run --rm -v vxllm-data:/data -v $(pwd):/backup \
  alpine tar czf /backup/vxllm-backup.tar.gz -C /data .
```

### Reset

```bash
# Stop containers and remove data
docker compose -f docker/docker-compose.yml down -v
```

## Production Recommendations

- **Set an API key** — Always set `API_KEY` when exposing VxLLM to the network.
- **Restrict CORS** — Set `CORS_ORIGINS` to your frontend domain instead of `*`.
- **Use a reverse proxy** — Place nginx or Caddy in front for TLS termination.
- **Monitor resources** — LLM inference is memory-intensive. Monitor RAM and VRAM usage.
- **Persistent storage** — Use a named volume (default) or a bind mount for model persistence.
